SenCadenas

Gestionnaire de mots de passe sécurisé — Projet de groupe · Module Programmation Sécurisée · 2025–2026

Django + DRF HTML · CSS · JavaScript JWT Auth SQLite NIST SP 800-63B Rev.4 Architecture Zero-Knowledge

SenCadenas est un gestionnaire de mots de passe sécurisé conçu pour le contexte sénégalais. Le projet répond à un besoin réel de cybersécurité en Afrique de l'Ouest, où la gestion des identifiants numériques reste souvent négligée. Il implémente une architecture Zero-Knowledge : les données sensibles ne sont jamais lisibles par le serveur, conformément aux recommandations du NIST SP 800-63B Rev.4 (2024).

Réalisé avec : Bachir Gueye  ·  Module Programmation Sécurisée  ·  École IPROSI

Statut du projet

✔ Frontend complet + Backend fonctionnel

Déployé en local · SQLite → PostgreSQL prévu

Architecture Générale

Le projet suit une architecture client-serveur avec séparation stricte des responsabilités entre le frontend Vanilla JS et l'API REST Django.

Couche Technologie Rôle
Frontend HTML5 · CSS3 · JavaScript (Vanilla) Interface utilisateur, interactions, chiffrement côté client
Backend Django 4.x + Django REST Framework API REST, logique métier, authentification
Base de données SQLite (développement) Stockage persistant des données chiffrées
Authentification JWT (djangorestframework-simplejwt) Sessions sécurisées, refresh tokens avec rotation

Frontend — 5 Pages HTML

Page d'accueil index.html

Présentation du projet avec sections Hero, Fonctionnalités, Comment ça marche. Aperçu animé du coffre-fort avec entrées simulées (Facebook, Wave, Sénégal Services…). Section sécurité avec présentation de l'architecture Zero-Knowledge. Navbar sticky avec scroll effect et menu burger responsive.

Inscription inscription.html

Formulaire avec validation en temps réel. Jauge de force du mot de passe basée sur l'entropie (conforme NIST). Minimum 15 caractères requis selon NIST SP 800-63B Rev.4. Appel API réel : POST /api/auth/register/ avec sauvegarde des tokens JWT.

Connexion login.html

Formulaire email + mot de passe maître. Blocage automatique après 5 tentatives échouées (2 minutes) avec compte à rebours visuel. Appel API réel : POST /api/auth/login/ avec récupération des tokens JWT.

Coffre-fort dashboard.html

Sidebar avec filtres (Tous, Favoris, Mots de passe, Cartes, Notes). Statistiques en temps réel : total, mots de passe faibles, réutilisés, forts. Score de sécurité global calculé dynamiquement. CRUD complet, copie en un clic, recherche et tri. Détection des mots de passe réutilisés dans le coffre.

Générateur NIST (intégré au dashboard)

Mode Phrase de passe : mots français aléatoires séparés (ex : Girafe-Soleil-Dakar-42). Mode Aléatoire : génération cryptographique via crypto.getRandomValues(). Calcul d'entropie en bits et estimation du temps de brute-force. Conforme NIST : longueur prioritaire sur la complexité.

Paramètres settings.html

Modification du profil (nom, email). Changement du mot de passe maître avec jauge NIST. Configuration 2FA TOTP avec QR code. Liste des sessions actives avec révocation. Zone de danger : export JSON, vidage du coffre, suppression de compte.

Protection contre les attaques

Type d'attaque Mesure Frontend Mesure Backend
Brute force Blocage après 5 tentatives (2 min) Throttling DRF : 20 req/heure (anon)
XSS Échappement avec escHtml() et escAttr() Django échappe automatiquement les templates
CSRF API stateless (pas de cookies de session) CsrfViewMiddleware + JWT Bearer auth
Injection SQL Validation côté client ORM Django (requêtes paramétrées)
Accès non autorisé Redirection si token absent IsAuthenticated + filtrage par user
Mots de passe faibles Jauge entropie + min 15 caractères MinimumLengthValidator (15 car.)

Autres mesures techniques

  • Tokens JWT stockés dans sessionStorage (effacés à la fermeture du navigateur)
  • Rafraîchissement automatique du token JWT expiré sans déconnecter l'utilisateur
  • Génération cryptographique des mots de passe avec crypto.getRandomValues() (pas Math.random())
  • Access token : durée de vie 30 min · Refresh token : 7 jours avec rotation et blacklist
  • Délégation d'événements pour éviter les conflits d'écouteurs lors du re-rendu du DOM

Stack Technologique

  • Frontend : HTML5 / CSS3 / JavaScript Vanilla — contrôle total sur la sécurité, pas de framework JS
  • Typographie : Syne + DM Sans + JetBrains Mono (Google Fonts)
  • Backend : Django 4.x + DRF — sécurité intégrée, ORM robuste, admin auto-généré
  • Auth : JWT (simplejwt) — stateless, sécurisé, standard industrie
  • BDD : SQLite (dév.) → PostgreSQL (production)
  • CORS : django-cors-headers
  • Standard : NIST SP 800-63B Rev.4

Perspectives d'Amélioration

  • Chiffrement côté client AES-256-GCM avant envoi au serveur (vraie architecture Zero-Knowledge)
  • Migration vers PostgreSQL pour la production
  • Intégration de l'API HaveIBeenPwned pour vérifier les mots de passe compromis
  • Implémentation complète du 2FA avec une vraie bibliothèque TOTP (pyotp)
  • Déploiement cloud (Heroku, Railway, ou VPS sénégalais)
  • Application mobile (React Native ou PWA)
  • Audit de sécurité complet et tests de pénétration
